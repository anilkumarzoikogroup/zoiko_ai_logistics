Phase 0

The Foundation Story

The Problem

Imagine Zoiko is a company that ships goods across the US.

They hired DHL to ship a package from Dallas to Atlanta.

The contract says DHL can charge $120 for fuel.

But DHL sent an invoice for $220 — charging an extra $100 for "accessorial fees" that were never authorized.

Someone needs to catch this, prove it, recover the $100 — and leave a paper trail no one can tamper with.

That's exactly what Phase 0 builds the foundation for.

What Phase 0 Built

Step 1 — The Invoice Arrives

DHL sends the invoice. The system receives it and immediately does 3 things:

  • Sorts and standardizes the invoice data the exact same way every time (JCS — like a universal translator for data)
  • Takes a fingerprint of it (SHA-256 hash) — like a wax seal on an envelope
  • Signs it with a digital signature — proof that Zoiko's system received exactly this document, untouched

Think of it like a notary stamping a document the moment it arrives.

Step 2 — The Database

25 tables were created to store every single event:

  • Who sent what
  • What was validated
  • Who approved what
  • What was executed
  • What was the final outcome

Think of it like filing cabinets — one drawer for invoices, one for approvals, one for audit records etc.

Step 3 — The Audit Trail (ACR)

At the end of the process, the system builds an Action Certification Record — a tamper-proof document that contains:

  • The original invoice fingerprint
  • The finding (confidence 0.96 — accessorial charge is unauthorized)
  • Both human approvals
  • The recovery execution
  • All locked together in a Merkle tree (like a chain of fingerprints)

If anyone changes even one word anywhere — the fingerprint changes and the tampering is instantly detected.

Step 4 — Offline Verification

The ACR can be given to any external auditor — a bank, a regulator, a court.

They don't need access to Zoiko's systems. They just run a verification script and it either says:

  • ✓ VERIFIED — $100 overcharge confirmed, recovery legitimate
  • ✗ TAMPERED — something was changed after signing

Like a sealed envelope with a wax stamp — if the seal is broken, everyone knows.

Phase 1

The Security Layer Story

The Problem Phase 0 Left Open

Phase 0 could fingerprint and sign documents perfectly. But it had no way to answer three critical questions:

  • Who is making this request? — Anyone could pretend to be Alice the analyst
  • Are they allowed to do this? — No rules existed to stop an analyst from approving their own proposal
  • How do services talk to each other safely? — No messaging system, no audit of which service said what

Phase 1 answers all three.

What Phase 1 Built

Component 1 — KMS Key Hierarchy (The Lock and Key System)

Every tenant (company using Zoiko) gets their own set of 3 dedicated keys:

  • Root CA Key — the master key, never used directly (like the vault key in a bank)
  • DEK Encrypt Key — encrypts the invoice data at rest (like a lockbox key)
  • Signing Key — signs every record to prove it came from that tenant (like a personal stamp)

In development: keys are generated in software on the local machine.

In production: keys never leave the GCP Cloud KMS hardware security module (HSM) — a tamper-proof chip. Even Zoiko engineers cannot extract the raw key material.

Think of it like a safety deposit box where only the bank's chip (not any person) holds the master key.

Component 2 — OIDC Identity (The Passport System)

Every API call into the system must carry a JWT token — a signed digital passport that proves:

  • Who you are (your email address)
  • Which company you belong to (tenant ID)
  • What role you have (analyst or manager)
  • When the passport expires (automatically rejected if expired)

Three things are tested every time:

  • Expired token → Rejected immediately
  • Tampered token → Rejected immediately
  • Wrong company → Rejected immediately

Think of it like airport security — no valid boarding pass, no entry. No exceptions.

Component 3 — OPA Policy Engine (The Rulebook Guard)

Even with a valid passport, you still need permission for what you're trying to do. OPA enforces the business rules:

  • Only an analyst can propose a recovery
  • Only a manager can approve it
  • The person who proposed cannot be the same person who approves — this is called Separation of Duties (SoD)
  • A company's analyst cannot touch another company's data — Tenant Isolation

Fail-Closed Rule: If the policy server is unreachable for any reason, the system returns an error and blocks the request. It never defaults to "allow".

Think of it like a bank teller who needs two managers to sign off on large withdrawals — and if the manager's phone is dead, the withdrawal stops, not proceeds.

Component 4 — Kafka Messaging (The Internal Postal System)

Instead of services calling each other directly, every event is published to a message queue. This means:

  • If one service is slow, others keep working
  • Every event is logged with a timestamp and tenant ID
  • No event can be lost — messages persist until consumed

17 registered event topics cover the full SC-001 lifecycle:

  • invoice.received → invoice.validated → invoice.canonical → case.opened → evidence.bundled → finding.created → proposal.created → decision.made → token.issued → execution.completed → acr.issued → and more

Think of it like a postal system inside the building — every department drops messages in a tray, and the right team picks them up. No department shouts across the room.

Phase 2

The Service Layer Story

What Phase 1 Left to Do

After Phase 1, the system knew who was making a request and whether they were allowed. But it still couldn't actually process a DHL invoice from end to end.

Phase 2 builds the four services that take a raw invoice and turn it into an open dispute case — ready for human review.

What Phase 2 Built

Service 1 — Ingestion Service (The Post Room)

When DHL sends an invoice for $220, the ingestion service handles it in 5 exact steps — always in this order, never skipped:

  • Step 1 — Standardize: The invoice fields are sorted and formatted identically every time (JCS). A $220 invoice will always produce the same bytes regardless of which server processes it.
  • Step 2 — Fingerprint: A SHA-256 hash of the standardized data is computed. This is the digital DNA of this invoice.
  • Step 3 — Encrypt: The invoice is encrypted before being stored. Only Zoiko's key can decrypt it.
  • Step 4 — Write to Database: Two records are saved in a single atomic transaction — the invoice record and an outbox entry. Either both save or neither does. No partial writes.
  • Step 5 — Announce on Kafka: After the database write is confirmed, the event invoice.received is published. Other services now know there's a new invoice to process.

Think of it like a post room that date-stamps every incoming letter, photographs it, seals it in a tamper-evident envelope, files a copy, and puts it in the outbox — all before anyone else touches it.

Service 2 — Validation Service (The Contract Checker)

This service reads the contract rates stored in the database and compares them against the invoice:

  • For DHL shipping Dallas → Atlanta, the contract says: $120 fuel charge
  • The invoice says: $220 total
  • The service calculates: $220 − $120 = $100 overcharge
  • Status: FAIL — carrier rate exceeded

The result is signed and saved. The event invoice.validated is published with the overcharge amount.

Three possible outcomes:

  • PASS — invoice matches contract, no problem
  • FAIL — invoice exceeds contract, overcharge detected
  • WARN — no contract rate found for this carrier, cannot verify

Think of it like a procurement officer who checks every supplier invoice against the signed purchase order before anything gets paid.

Service 3 — Canonical Truth Service (The Single Source of Truth)

After validation, the invoice needs one definitive home in the database — a single row that all future services treat as the ground truth.

This service writes:

  • A canonical_invoice row — the official version of the invoice (carrier, amount, currency, invoice number)
  • A canonical_shipment row — the route details (Dallas → Atlanta, 1,200 lbs)
  • A canonical_hash — a new fingerprint computed over the canonical form. Every downstream service — evidence gathering, AI reasoning, governance approval, the final audit record — will reference this exact hash.

If the same invoice is submitted twice, the service returns the same record — no duplicates ever created.

Think of it like a court clerk creating the official certified copy of a document — from this point on, everyone references the clerk's copy, not the original fax.

Service 4 — Case Orchestration Service (The Case Manager)

Now that there's a validated, canonical invoice showing a $100 overcharge, a formal dispute case is opened.

The case follows a strict state machine — it can only move forward in defined steps:

StateMeaning
OPENEDCase registered, dispute formally started
EVIDENCE GATHERINGSystem collects all proof (Phase 3)
UNDER REVIEWAI reasoning service analyzes the evidence (Phase 3)
PENDING APPROVALWaiting for analyst + manager sign-off (Phase 3)
APPROVEDBoth humans have signed
EXECUTEDRecovery payment sent to carrier (Phase 4)
RECONCILEDPayment confirmed and matched
CLOSEDCase complete, ACR issued

Every state change is written as an append-only case event — nothing is ever updated or deleted. The complete history of who moved the case, when, and why is permanently preserved.

At the end of Phase 2, the SC-001 case is at PENDING APPROVAL — waiting for Alice (analyst) and Bob (manager) to review the $100 overcharge finding.

Think of it like a legal case file — every action is stamped, signed, and added to the file. The old entries are never crossed out.

Phase 0 Kicks In — The Moment the Invoice Arrives

BlueDart uploads invoice BD-HYD-WGL-20260518 into the Zoiko portal. Before any human sees it, the Phase 0 crypto layer immediately runs three operations.

Operation 1 — JCS Canonicalization

The invoice fields are sorted alphabetically by Unicode code point and serialised into a single deterministic UTF-8 byte string:

Raw field from BlueDartJCS-sorted positionValue
carrier_id1st"BlueDart"
currency2nd"INR"
invoice_number3rd"BD-HYD-WGL-20260518"
route_destination4th"Warangal"
route_origin5th"Hyderabad"
total_amount6th"12500.0"

Code that runs: zoiko_common.crypto.jcs.canonicalize(payload_dict) — same result every time on any server worldwide.

Operation 2 — Domain-Tagged SHA-256 Fingerprint

The system prepends a domain tag and computes the fingerprint:

canonical_hash = SHA-256( b"zoiko.ingestion.invoice.v1:" + canonical_bytes )

This produces a unique 64-character hex string — the digital DNA of BlueDart's ₹12,500 invoice. If BlueDart later claims they sent ₹8,000, the hash will not match. The lie is mathematically impossible to hide.

Operation 3 — KMS Signing

The hash is signed with Amazon India's dedicated signing key stored in the KMS key hierarchy. The signing key is tenant-specific — BlueDart's signature cannot be forged using Flipkart's key, and Flipkart's key is never even loaded during this operation.

Code that runs: shared.signer.sign("amazon-india", canonical_hash)

Phase 3

The Human Decision Layer Story

Where Phase 2 Left Off

After Phase 2, the system had a formally opened dispute case with a clear finding: BlueDart charged Amazon India ₹12,500 but the contract only allows ₹8,000 — a ₹4,500 overcharge. The case was sitting in state PENDING_APPROVAL.

But no money had moved. No human had reviewed the evidence. No one had signed off. Phase 3 fixes all of that.

Phase 3 answers four questions:

  • What proof exists? — Evidence Service collects and locks all supporting documents
  • How confident is the system? — Reasoning Service computes a deterministic 96% confidence score
  • Did two different humans approve? — Governance Service enforces Separation of Duties
  • What unlocks execution? — Token Service mints a one-time cryptographic key tied to this exact decision

What Phase 3 Built

Service 1 — Evidence Service (The Evidence Locker)

Ravi uploads three documents to prove the overcharge:

DocumentWhat it contains
Bill of Lading (BOL)800 kg electronics, Hyderabad depot, 15 May 2026
Rate SheetSigned contract: BlueDart to charge ₹8,000 for Hyderabad→Warangal 800 kg
InvoiceBlueDart invoice BD-2026-0512: ₹12,500 charged for the same shipment

For each document, the service runs two operations:

  • Domain-tagged hash: SHA-256("zoiko.evidence.item.v1:" + document_bytes) — the domain prefix means a tampered invoice cannot be passed off as a Bill of Lading
  • Merkle tree update: After each item is added, the Merkle root over all items is recomputed and stored in the evidence bundle

After 3 items, the bundle holds a single Merkle root — a 64-character hash that represents all three documents simultaneously. If anyone changes even one byte in any document, the root changes, and the tampering is immediately visible.

Think of it like a sealed evidence box in a court case. Once the lid is sealed and stamped, you can see exactly which items are inside — and if anything is added or removed, the seal breaks.

Service 2 — Reasoning Service (The AI Analyst)

With the evidence bundle locked, the Reasoning Service computes a confidence score for the overcharge finding. The formula is deterministic — it always produces the same answer for the same inputs, with no randomness and no model variation:

RuleConfidenceWeightContribution
Fuel charge check1.00 (exact contract match)50%0.50
Accessorial charge check0.92 (surcharge not in contract)50%0.46
Total confidence0.96100%0.96

The service then creates two records:

  • Finding — a signed record of the confidence score and rule trace, anchored to the evidence bundle hash
  • Decision Proposal — Ravi's formal request: "Issue a CREDIT_MEMO for ₹4,500 to recover the overcharge"

Both records are JCS-canonicalized, domain-tagged SHA-256 hashed, and Ed25519-signed — so the finding cannot be altered after Ravi submits it, and Ramu knows the 0.96 score he's seeing is exactly what the system computed.

Think of it like a forensic accountant's report — a numbered, signed document that says "I examined the evidence, and I am 96% confident this is an overcharge."

Service 3 — Governance Service (The Two-Key Safe)

Now the proposal needs human approval. Phase 3 enforces a rule that cannot be bypassed in code: the person who proposed the recovery cannot be the same person who approves it.

This is called Separation of Duties (SoD).

Step 1 — Ravi creates an approval task:

  • An approval_task record is created: status=PENDING, proposer_sub=ravi@amazon.in
  • The task is published on Kafka. Ramu's notification queue lights up.

Step 2 — Ravi tries to approve his own proposal:

  • Ravi submits himself as the approver
  • The service checks: actor_sub ('ravi@amazon.in') == proposer_sub ('ravi@amazon.in')
  • Result: ValueError raised before any database write occurs — "Separation of Duties violation: actor_sub 'ravi@amazon.in' cannot be the same as proposer_sub 'ravi@amazon.in'"
  • Nothing is written. Nothing is published. The attempt leaves no trace.

Step 3 — Ramu approves (SoD passes):

  • Ramu submits his approval: actor_sub=ramu@amazon.in
  • SoD check passes (ramu ≠ ravi)
  • The decision payload is JCS-canonicalized and signed: SHA-256("zoiko.governance.decision.v1:" + decision_bytes)
  • A governance_decision record is written with outcome=APPROVED
  • The case state machine transitions: PENDING_APPROVAL → APPROVED
  • An APPEND-ONLY case_event is written: "GOVERNANCE_APPROVED — ramu@amazon.in"
  • Kafka: decision.made published

Think of it like a nuclear launch: two keys, in two different hands, both must turn — and neither person can use both keys.

Service 4 — Token Service (The One-Time Key)

With the governance decision recorded and the case in APPROVED state, the Token Service mints a cryptographic execution token. This token is the only key that can unlock the Phase 4 Execution Gateway — and it can only be used once.

signatureEd25519 sign(token_hash) using tenant's signing keyForgery is mathematically impossible
expires_atnow + 24 hoursStolen tokens expire automatically
statusACTIVE → CONSUMED (Phase 4 sets this atomically)No replay attacks — token cannot be reused after execution

The scope field is set to EXECUTE_CREDIT_MEMO — the token can only be used to issue a credit memo. It cannot be used to execute a wire transfer or any other action.

Think of it like a time-limited, single-use key card that opens one specific door — and burns itself the moment it's used.

Phase 5

The Frontend Story — Putting a Human Face on the Pipeline

The Problem Phases 0–3 Left Open

After Phase 3, the entire overcharge detection and recovery pipeline existed — but only as APIs and command-line demos. Ravi (analyst) and Ramu (manager) had no screen to look at. There was no way to submit an invoice from a browser, no queue showing which cases needed review, and no dashboard showing how much money had been recovered.

Phase 5 builds the React + TypeScript frontend that makes the entire system usable by real humans — without changing a single line of backend code.

What Phase 5 Built

The Live Pipeline Monitor (Home)

The first screen any user sees shows the pipeline as it runs in real time:

  • How many invoices were ingested today, how many are in each stage
  • KPI cards: total recovered, cases pending approval, avg latency
  • A live table of "invoices in flight" — each one linked to its case detail page
  • Action banners: "3 cases awaiting analyst review" with a direct link

Think of it like an air traffic control screen — every invoice is a plane, and the controller can see exactly where each one is in the sky.

Submit Invoice (New Case)

Ravi can submit a new invoice in two ways:

  • Upload a PDF — the backend parses it with pdfplumber, extracts carrier name, amount, currency, and route automatically. Fields appear pre-filled.
  • Enter manually — pick carrier, origin city, destination city, amount, currency from dropdowns.

When Ravi clicks "Submit for Audit", the frontend calls the Phase 2 API which runs the full pipeline — ingestion, validation, canonical truth, case open, evidence bundling, and AI reasoning — all in one request. A green toast notification confirms: "Case submitted — overcharge detection pipeline started."

Ravi is immediately redirected to the new case's detail page, already showing the AI confidence score and which pipeline stages have completed.

Case Detail — Where Is This Case?

Every case has a detail page with an 8-stage pipeline stepper:

1. Invoice IngestedSHA-256 domain-tagged hash, source record ID

At the bottom, a full event timeline shows every state transition: who made it, when, and why — in chronological order.

Analyst Review Queue

Ravi opens the Analyst Review page and sees all cases in NEW, EVIDENCE_PENDING, or FINDING_GENERATED state — sorted by AI confidence, highest first. For each case he sees:

  • The AI confidence score (96%, colour-coded green if ≥ 90%)
  • Carrier name and shipment reference
  • Invoice amount and overcharge amount
  • A "Propose ₹4,500" button

When Ravi clicks Propose, the frontend calls POST /cases/{id}/proposal. The case moves to APPROVAL_PENDING. The sidebar badge on "Manager Approval" increments by 1 — updated automatically every 30 seconds by background polling.

A green toast confirms: "Recovery proposed — ₹4,500 sent to manager approval."

Manager Approval Queue

Ramu opens the Manager Approval page and sees one card for the BlueDart case. He reads the AI confidence, the proposed amount, and the analyst's name. He has two buttons: Approve and Reject.

If Ramu tries to approve a case he himself proposed, the API rejects it before any DB write — and the frontend shows a red toast: "Decision failed — check SoD rule, you cannot approve your own proposal."

When Ramu approves legitimately, the case moves to EXECUTION_READY, a governance token is issued, and a green toast confirms: "Case approved — governance token issued, 15-min execution window open."

Analytics & Trends

The Analytics page connects to real data from the cases API:

  • KPI cards — recovery rate, avg overcharge, total recovered — computed live from the cases array
  • Recovery trend bar chart — groups real cases by date opened, shows cases and amounts recovered day by day
  • Carrier scorecard — counts cases per carrier from live data, computes avg overcharge per carrier

When the database is empty (fresh install), all charts fall back to demo data with a yellow badge: "Demo data — submit invoices to see live charts."
